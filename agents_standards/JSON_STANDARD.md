# JSON STANDARD

## 基本要求
- 所有 `JSON` 文件必须使用 `UTF-8` 编码；
- 所有 `JSON` 文件必须是合法标准 `JSON`，不得使用非标准扩展语法；
- 禁止使用注释，因为标准 `JSON` 不支持注释；
- 禁止使用尾随逗号；
- 禁止使用单引号，字符串必须使用双引号；
- 所有键名必须是字符串；
- 所有结构必须保证可被标准 `JSON` 解析器直接解析；
- 文档内容必须优先保证清晰、稳定、可维护、可解析；

## 结构要求
- 顶层结构必须固定，并与业务语义保持一致；
- 同一类文件的顶层结构必须统一，例如统一为对象或统一为数组；
- 若无特殊理由，顶层结构优先使用对象；
- 所有对象结构必须层级清晰，禁止无节制嵌套；
- 数据层级应尽量扁平，避免过深嵌套影响可读性和维护性；
- 固定语义的数据必须使用固定字段，不得随意漂移；
- 相同语义的数据在不同位置必须保持相同结构；
- 空结构的使用必须有明确规则，例如空对象 `{}` 或空数组 `[]` 的含义必须稳定；

## 键名规范
- 所有键名必须语义明确，禁止使用无意义键名，例如 `a`、`b`、`tmp`、`data1`；
- 同一项目中的键名风格必须统一；
- 若无特殊约定，推荐统一使用 `snake_case` 或 `camelCase`，不得混用；
- 同一语义只能使用一种命名，不得出现同义字段并存；
- 禁止随意修改既有键名；
- 键名必须稳定，避免因命名变化导致兼容性问题；
- 布尔字段应采用可表达真假语义的命名，例如 `is_enabled`、`has_error`、`can_retry`；
- 列表字段名称应体现集合语义；
- 时间、状态、类型、标识等高频字段必须统一命名；

## 值类型要求
- 每个字段的值类型必须稳定；
- 同一个字段不得在不同记录中随意变化类型；
- 数值字段必须明确区分整数和浮点数语义；
- 布尔字段必须使用标准布尔值 `true` 或 `false`；
- 空值只能使用 `null` 表达；
- 禁止用空字符串、`0`、`false`、空数组替代 `null`，除非规范明确允许；
- 枚举字段必须限定允许值范围；
- 标识字段必须保证唯一性或唯一性范围明确；
- 时间字段格式必须统一，禁止同一文件内混用多种时间格式；

## 数组要求
- 数组中的元素类型必须一致；
- 同一数组中不得混合多种无关结构；
- 若数组元素为对象，则对象字段集合应尽量一致；
- 数组顺序若有业务含义，必须稳定；
- 若数组顺序无业务含义，应在规范中明确；
- 空数组必须具有明确语义，不得滥用；
- 大数组应考虑可拆分、分页或流式处理需求；
- 禁止将不相关的数据强行塞入同一数组；

## 对象要求
- 对象字段必须职责清晰；
- 对象中不得塞入与其语义无关的字段；
- 固定结构对象必须保持字段集合稳定；
- 字段是否必填必须有明确约束；
- 必填字段不得缺失；
- 可选字段是否允许缺省、是否允许 `null` 必须定义清楚；
- 对象中的嵌套对象必须有明确语义边界；
- 不得使用超宽对象承载过多无关信息；

## 可读性要求
- 文件排版必须统一；
- 推荐统一使用 `2` 个或 `4` 个空格缩进，项目内不得混用；
- 同一类文件的字段顺序应尽量保持稳定；
- 高优先级字段应放在靠前位置；
- 相同结构对象的字段顺序应尽量一致；
- 长文件应保持逻辑分组清晰；
- 不得为了压缩体积而牺牲源文件可读性，除非该文件明确为构建产物；
- 结构复杂时，应配套提供独立规范或 schema 说明；

## 兼容性要求
- 已投入使用的字段必须尽量保持向后兼容；
- 新增字段应优先采用可选方式扩展；
- 删除字段前必须评估兼容性影响；
- 修改字段语义必须视为高风险变更；
- 不得在无迁移方案的情况下重命名已有字段；
- 枚举值扩展必须考虑旧版本消费者行为；
- 输入输出格式必须稳定，避免隐式字段漂移；
- 向后不兼容变更必须有明确记录和迁移说明；

## Schema 与校验
- 重要 `JSON` 文件必须有明确的结构约束；
- 推荐为核心结构提供 `JSON Schema` 或等价校验规则；
- 所有自动化依赖字段都必须可稳定校验；
- 必填字段、字段类型、枚举值、默认行为必须可定义；
- 校验规则必须与实际数据保持同步；
- 校验失败时必须给出明确、可定位的错误信息；
- 结构变更后必须同步更新 schema 或对应文档；
- 生成类文件和手写类文件的校验要求应保持一致；

## 默认值与空值
- 默认值策略必须统一；
- 若字段可缺省，则必须明确缺省语义；
- 若字段允许 `null`，则必须明确 `null` 的业务含义；
- 禁止同时使用“字段缺失”和“字段为 `null`”表达同一种含义，除非规范明确区分；
- 默认值应选择安全、可预测、可解释的值；
- 布尔字段必须避免三态不清问题；
- 空字符串只应用于明确需要空文本的场景；
- 空数组和空对象必须保留其结构语义，不得随意替代其他空值表达；

## 时间与标识
- 时间字段必须统一格式；
- 若无特殊要求，推荐使用 `ISO 8601` 格式；
- 时间字段必须明确时区语义；
- 同一文件中不得混用本地时间和 UTC 时间而不加说明；
- 标识字段命名必须统一，例如统一使用 `id`、`name`、`type`、`status`；
- 标识值必须保持稳定，禁止将展示文案当作唯一标识；
- 若存在主键、外键、引用关系，必须保证引用可追踪；
- 状态字段必须限定在预定义集合内；

## 安全要求
- 禁止在 `JSON` 文件中明文存储密码、私钥、密钥、令牌等敏感信息；
- 禁止提交包含生产环境凭据的 `JSON` 文件；
- 日志、导出、调试生成的 `JSON` 不得泄露敏感字段；
- 外部输入生成的 `JSON` 必须经过合法性校验；
- 反序列化后的字段必须经过业务层校验后再使用；
- 不可信来源的数据不得直接写入核心配置；
- 任何敏感字段都必须有脱敏、隔离或受控策略；
- 测试数据中的敏感字段必须使用伪造值；

## 配置文件要求
- 作为配置用途的 `JSON` 文件必须保持简洁；
- 配置项命名必须清晰、稳定、可预期；
- 可变配置不得与运行时生成数据混写；
- 配置文件中的默认值必须安全；
- 配置项优先体现业务含义，而不是底层实现细节；
- 配置项之间的依赖关系必须明确；
- 高风险配置必须有显式开关或额外约束；
- 配置字段变更必须同步更新使用方；

## 数据文件要求
- 作为数据存储用途的 `JSON` 文件必须保证结构一致；
- 批量记录中的对象字段集合应尽量统一；
- 数据文件不得混入无关元信息，除非规范明确要求；
- 统计类、缓存类、快照类文件必须在语义上区分；
- 生成数据与手工维护数据应尽量分离；
- 大体积数据必须考虑拆分策略；
- 数据更新策略必须明确，例如全量覆盖、增量追加、按键替换；
- 删除数据应有明确策略，不得静默丢失关键记录；

## 日志与输出要求
- 面向自动化消费的 `JSON` 输出必须字段稳定；
- 日志型 `JSON` 必须包含足够上下文以支持排查；
- 输出字段应避免混入仅供展示的人类文本；
- 关键状态、错误信息、上下文标识应结构化表达；
- 错误输出必须可定位；
- 重要输出优先使用可解析对象，而不是拼接字符串；
- 字段扩展必须考虑现有消费者兼容性；
- 不同输出类型应有清晰边界，避免混合语义；

## 生成与维护要求
- 自动生成的 `JSON` 文件应与手工维护文件明确区分；
- 自动生成文件若不应手改，必须有明确标识；
- 手工维护文件必须保持格式稳定，减少无意义 diff；
- 字段顺序、缩进、换行风格应尽量固定；
- 变更时必须控制影响范围；
- 核心结构变更必须同步更新消费方、校验规则和说明文档；
- 不得静默变更关键字段语义；
- 历史兼容性要求高的文件必须有变更审计机制；

## 性能与体积要求
- 不得在单一 `JSON` 文件中无限堆积大体积数据；
- 高频读写文件必须考虑解析成本；
- 大文件应评估拆分、索引、分页或二进制替代方案；
- 不得在配置类文件中嵌入大块无关数据；
- 同一份数据不得无意义重复冗余；
- 对体积敏感场景可进行压缩，但源文件规范仍应保持可维护；
- 结构设计应兼顾传输成本和清晰表达；
- 性能优化不得破坏结构稳定性；

## 测试与校验要求
- 重要 `JSON` 文件必须具备最小校验路径；
- 至少应覆盖合法样例、非法样例、边界样例；
- 对关键配置和核心数据应提供自动化验证；
- 校验失败时应能定位到字段级别；
- 变更后必须验证旧数据兼容性；
- 依赖 `JSON` 的核心逻辑必须有回归测试；
- 示例数据应尽量贴近真实使用场景；
- 测试数据不得污染正式数据结构；

## 禁止事项
- 禁止使用非标准 `JSON` 语法；
- 禁止使用注释；
- 禁止使用尾随逗号；
- 禁止混用多套键名风格；
- 禁止同一字段在不同地方语义漂移；
- 禁止用展示文案替代结构化字段；
- 禁止在敏感场景中明文存储凭据；
- 禁止无文档、无校验地随意扩展核心结构；
- 禁止将缺失、`null`、空字符串、空数组混为一谈；
- 禁止在单个字段中塞入多个未拆分语义；
