# BATCH STANDARD

## 基本要求
- 所有 `BAT` 文件必须使用 `UTF-8` 编码，并确保目标执行环境可正确识别编码；
- 所有脚本必须优先保证可读性、可维护性、可审计性和可回滚性；
- 同一项目中的脚本风格、变量风格、错误处理风格、日志风格必须保持一致；
- 禁止为了省事堆叠不可读的单行命令；
- 禁止引入与当前需求无关的复杂技巧；
- 所有脚本必须在 Windows 标准命令解释器环境下可运行；
- 若脚本依赖特定系统版本、特定工具链或特定环境变量，必须显式说明；

## 文件与结构要求
- 所有脚本必须以 `@echo off` 开头；
- 建议在开头使用 `setlocal` 控制变量作用域；
- 若脚本使用延迟扩展，必须显式使用 `setlocal EnableDelayedExpansion`；
- 脚本开头必须包含脚本用途说明；
- 脚本主体结构必须清晰，初始化、参数解析、核心流程、清理收尾应有明确边界；
- 复杂脚本应拆分为多个标签段，例如 `:main`、`:parse_args`、`:cleanup`；
- 标签命名必须语义明确，禁止使用无含义短名；
- 脚本结束时必须通过显式 `exit /b` 返回退出码；
- 禁止依赖命令自然结束作为正式退出方式；

## 命名规范
- 环境变量命名必须语义明确；
- 项目级变量建议使用统一前缀，避免与系统变量冲突；
- 变量名建议使用全大写加下划线风格，例如 `OUTPUT_DIR`、`EXIT_CODE`；
- 临时变量必须体现用途，禁止使用 `a`、`b`、`tmp1`、`xx` 这类无语义名称；
- 标签名必须体现动作或职责，例如 `:copy_files`、`:validate_input`；
- 布尔语义变量必须尽量清晰，例如 `HAS_ERROR`、`DRY_RUN`、`VERBOSE_MODE`；

## 注释规范
- 注释必须解释原因、边界、约束和非直观行为，而不是复述命令表面含义；
- 简单直白的命令不得强行添加重复性注释；
- 对复杂跳转、参数约束、外部依赖、危险操作、失败回退必须注释清楚；
- 注释必须与脚本实现保持同步；
- 过期注释必须及时删除或更新；
- 禁止写无信息量注释，例如“设置变量”“执行命令”；
- 禁止使用情绪化、口语化或个人化注释；
- 临时说明应使用统一标记，例如 `REM TODO:`、`REM FIXME:`，并附带明确原因；

## 代码风格
- 一条命令只做一个清晰动作；
- 复杂逻辑必须拆分为多个步骤，不得堆叠成长链命令；
- 条件判断、循环、跳转必须尽量保持浅层嵌套；
- 禁止滥用 `goto` 形成不可追踪流程；
- 允许使用标签和 `call` 组织流程，但必须保证调用路径清晰；
- 核心流程必须可追踪、可调试、可定位；
- 危险命令前必须显式做条件检查；
- 相同逻辑必须抽取复用，禁止大段复制粘贴；
- 命令书写风格必须统一，例如统一使用 `if not exist` 或统一使用正向判断；
- 对带空格路径、带特殊字符参数的处理必须始终保持一致；

## 变量与扩展
- 所有变量赋值必须使用 `set "KEY=value"` 风格；
- 禁止使用裸 `set KEY=value` 以避免尾随空格和特殊字符问题；
- 需要在循环或代码块中读取动态变量时，必须明确使用延迟扩展；
- 使用延迟扩展时必须注意 `!` 字符带来的内容破坏问题；
- 脚本中不得混乱使用 `%VAR%` 和 `!VAR!`；
- 变量作用域必须清晰，避免无意污染调用环境；
- 临时变量使用后应及时清理或限定在 `setlocal` 生命周期内；
- 重要变量必须在使用前初始化；

## 参数处理
- 脚本必须清晰定义支持的参数格式；
- 命令行参数必须提供帮助说明入口，例如支持 `-h`、`--help`、`/help` 中的一种或多种；
- 参数名称和行为必须稳定，禁止隐式漂移；
- 参数解析逻辑必须集中处理，避免散落在主流程中；
- 必填参数缺失时必须返回非零退出码并输出清晰错误信息；
- 可选参数必须有明确默认值；
- 未识别参数必须显式报错，不得静默忽略；
- 参数值若涉及路径、模式、环境名、标识名等，必须做合法性校验；
- 布尔参数必须有统一风格，例如统一采用 `--dry-run` 或 `/dryrun` 风格；

## 路径与文件处理
- 所有路径处理必须考虑空格路径情况；
- 路径变量在使用时必须始终使用引号包裹；
- 路径拼接必须清晰稳定，禁止依赖脆弱字符串拼接；
- 文件读写、复制、移动、删除前必须检查目标存在性和预期状态；
- 涉及覆盖写入时必须明确覆盖行为；
- 涉及删除操作时必须增加保护条件，避免误删；
- 临时文件和临时目录必须有明确生命周期；
- 跨盘符操作、相对路径解析和当前工作目录变更必须谨慎处理；
- 若脚本依赖自身所在目录，必须使用 `%~dp0` 或等价方式显式定位；
- 禁止假定脚本总是在固定工作目录下执行；

## 错误处理
- 关键命令执行后必须显式检查失败状态；
- 对关键路径必须检查 `errorlevel` 或等价失败信号；
- 禁止在失败后继续无条件执行后续高风险步骤；
- 报错信息必须包含足够上下文，至少说明失败动作和目标对象；
- 禁止吞掉错误而不输出任何信息；
- 可确定修复的失败应明确区分为确定性修复；
- 非确定性失败不得伪装为成功；
- 对外部命令、文件系统操作、网络访问、依赖工具调用必须显式处理失败分支；
- 所有失败出口必须返回非零退出码；
- 成功路径必须明确返回 `0`；

## 日志与输出
- 脚本输出必须清晰区分正常信息、警告信息和错误信息；
- 关键步骤必须有可读日志输出；
- 日志内容必须简洁明确，避免输出无意义噪声；
- 错误日志必须帮助快速定位问题；
- 涉及自动化集成时，重要输出应尽量结构化或至少格式稳定；
- 禁止把正式日志完全依赖于调试输出；
- 在批量循环处理中，日志粒度必须适中，避免刷屏；
- 成功、失败、跳过、重试等状态建议使用统一前缀；
- 输出中不得泄露敏感信息，例如密码、令牌、密钥、内部地址凭据等；

## 流程控制
- 主流程入口必须清晰，建议显式跳转到 `:main`；
- 初始化逻辑必须在主流程前完成；
- 参数解析、环境校验、核心执行、清理收尾应保持顺序清晰；
- 所有 `goto` 跳转目标必须明确且有限；
- 禁止构造复杂不可读的多层跳转网络；
- 公共逻辑建议通过 `call :label` 方式复用；
- 子流程必须通过显式返回保持调用边界；
- 清理逻辑必须在成功和失败两条路径下都可执行；

## 外部命令与依赖
- 所有外部命令调用前必须确认依赖存在；
- 缺少依赖时必须输出清晰提示；
- 外部命令路径若非系统默认可用，必须显式配置；
- 外部命令调用必须检查返回码；
- 命令调用涉及参数拼接时必须防止特殊字符破坏；
- 若脚本依赖 `PowerShell`、`Python`、`Git`、`7z` 等工具，必须写明最低可接受环境；
- 对高风险命令调用，必须增加前置校验和失败回退；

## 兼容性要求
- 脚本必须说明目标适配的 Windows 环境范围；
- 不同系统区域设置、编码环境、命令解释差异必须提前考虑；
- 涉及中文路径、空格路径、特殊字符路径时必须测试；
- 使用延迟扩展、管道、重定向、循环嵌套时必须考虑 `cmd` 特性差异；
- 若某写法仅适用于特定版本系统或外壳行为，必须显式注明；
- 禁止默认假设执行环境完全一致；

## 安全要求
- 禁止硬编码密码、令牌、密钥和敏感凭据；
- 删除、覆盖、移动、格式化等破坏性操作必须显式受控；
- 涉及系统目录、用户目录、环境变量修改时必须谨慎处理；
- 从外部输入拼接命令时必须做必要校验；
- 禁止无保护执行用户可控输入形成的命令；
- 日志和错误输出中不得泄露敏感信息；
- 默认行为必须偏安全，必要时要求显式确认；
- 危险模式建议增加 `dry-run` 或预览能力；

## 测试与校验
- 每个脚本至少应提供最小可执行验证路径；
- 关键脚本必须具备最小成功路径和失败路径校验；
- 参数缺失、路径不存在、依赖缺失、命令失败等场景必须可测试；
- 关键命令必须支持非零退出码；
- 重要输出格式必须稳定，方便自动化消费；
- 回归问题应补充对应测试样例或最小复现用例；
- 测试不得依赖不可控外部状态；
- 生产脚本与测试脚本必须职责分离；

## 可维护性要求
- 脚本职责必须单一，过大的脚本必须拆分；
- 通用逻辑应沉淀为可复用子脚本或标签段；
- 修改脚本时必须控制影响范围；
- 大改动必须分步骤实施，避免一次性引入大量不确定性；
- 所有关键实现都应做到可定位、可回滚、可审计；
- 临时兼容代码必须附带清理计划；
- 技术债必须可见，不得隐藏在复杂跳转和魔法变量中；
- 对长期维护脚本，必须提供使用说明和参数说明；

## 禁止事项
- 禁止缺失 `@echo off`；
- 禁止裸 `set KEY=value`；
- 禁止未加引号直接使用可能包含空格的路径；
- 禁止不检查 `errorlevel` 就继续执行关键步骤；
- 禁止空错误处理；
- 禁止吞掉失败状态；
- 禁止依赖隐式当前目录；
- 禁止滥用 `goto` 制造不可维护流程；
- 禁止硬编码敏感信息；
- 禁止无保护执行高风险删除或覆盖命令；
- 禁止长期保留调试代码、临时跳过逻辑和无效注释；
