# PYTHON STANDARD

## 基本要求
- 所有 Python 文件必须使用 `UTF-8` 编码；
- 默认使用 `Python 3.10+` 语法特性时，必须确保项目运行环境已明确支持；
- 所有代码必须优先保证可读性、可维护性和可测试性；
- 同一项目中的代码风格、命名风格、异常风格、日志风格必须保持一致；
- 禁止为了炫技使用不必要的高级语法；
- 禁止引入与当前需求无关的复杂抽象；
- 代码实现必须优先追求清晰，再考虑过早优化；

## 代码风格
- 默认遵循 `PEP 8`；
- 行宽应保持稳定，建议不超过 `88` 或项目统一约定的最大长度；
- 函数应尽量短小，单个函数只负责一个清晰职责；
- 复杂流程必须拆分为多个小函数，不得将多个职责堆叠在一个函数中；
- 条件分支、循环、异常处理嵌套层级不得过深；
- 优先使用显式写法，避免可读性差的链式表达；
- 避免隐式副作用，核心流程必须保持可追踪；
- 禁止滥用全局变量；
- 禁止在模块导入阶段执行高副作用逻辑；
- 所有常量必须集中定义，禁止在代码中散落魔法值；
- 重复逻辑必须提取复用，禁止无意义复制粘贴；

## 命名规范
- 变量名、函数名、模块名统一使用 `snake_case`；
- 类名统一使用 `PascalCase`；
- 常量名统一使用 `UPPER_SNAKE_CASE`；
- 私有辅助函数或内部属性使用前缀下划线，例如 `_internal_func`；
- 命名必须语义明确，禁止使用 `tmp`、`foo`、`bar`、`data2`、`test1` 这类无业务含义名称；
- 布尔变量应使用可直接表达真假语义的命名，例如 `is_enabled`、`has_error`、`can_retry`；
- 集合、列表、映射等命名应体现复数或容器语义；
- 函数名应体现动作，变量名应体现数据；

## 类型标注
- 公共函数、公共方法、核心内部函数必须显式提供类型标注；
- 函数参数和返回值必须尽量完整标注；
- 对外暴露的接口不得省略返回类型；
- 优先使用标准库 `typing` 中的稳定类型表达；
- 复杂嵌套类型应通过类型别名简化可读性；
- 当返回值结构固定时，优先使用明确的数据结构定义，而非松散 `dict`；
- 类型标注必须服务于可读性，禁止堆砌过度复杂的类型表达；
- 允许在动态场景下使用 `Any`，但必须控制范围并说明原因；

## 函数设计
- 函数必须职责单一；
- 函数参数数量应尽量精简，过多参数必须考虑封装；
- 优先使用显式参数，避免依赖隐式上下文；
- 有默认值的参数必须选择安全、可预测的默认行为；
- 禁止使用可变对象作为默认参数，例如 `[]`、`{}`；
- 函数返回值必须稳定，不得在不同分支返回语义差异过大的类型；
- 对外函数的输入输出必须可预期；
- 核心函数应尽量保持无副作用或副作用可控；
- 函数过长、分支过多、职责混杂时必须拆分；

## 类设计
- 只有在确实需要封装状态或行为聚合时才使用类；
- 简单数据承载优先考虑 `dataclass` 或等价轻量结构；
- 类必须有明确职责边界，禁止上帝类；
- 公共属性与内部属性必须边界清晰；
- 继承必须谨慎使用，优先组合而非深层继承；
- 抽象基类、接口类只在确有复用和约束价值时引入；
- 类方法、静态方法、实例方法的使用必须符合职责语义；
- 初始化逻辑不得过于复杂，复杂初始化必须拆分；

## 注释规范
- 注释必须解释原因、约束、边界和非直观点，而不是复述代码表面含义；
- 简单直白的代码不得强行添加重复性注释；
- 对复杂逻辑、特殊分支、外部依赖、边界条件、失败回退必须写清楚；
- 注释必须与代码保持同步，过期注释必须及时删除或更新；
- 禁止写无信息量注释，例如“给变量赋值”“循环处理数据”；
- 禁止使用情绪化、口语化、个人化注释；
- 模块头注释可用于说明模块职责、输入输出和使用限制；
- 函数注释应优先说明用途、关键参数、返回值语义和异常约束；
- 临时性说明不得长期保留，`TODO`、`FIXME`、`HACK` 必须带原因或后续处理方向；

## 文档字符串
- 对外模块、公共类、公共函数建议提供 `docstring`；
- `docstring` 必须说明功能目的，而不是逐行翻译实现；
- 参数较多或行为复杂的函数必须说明参数含义；
- 返回值语义不明显时必须说明返回内容；
- 会抛出重要异常的函数应说明异常场景；
- 文档字符串风格应在项目内统一，例如统一采用 `Google` 风格或简洁风格；
- 同一项目中不得混用多套完全不同的文档字符串风格；

## 导入规范
- 导入顺序必须稳定，通常按标准库、第三方库、本地模块分组；
- 禁止使用 `from xxx import *`；
- 禁止出现未使用导入；
- 循环依赖必须避免；
- 模块路径必须清晰，禁止通过脆弱相对路径 hack 导入；
- 导入别名只在缩短高频且公认的名称时使用；
- 模块导入必须保持顶部声明，除非延迟导入能明确解决性能或循环依赖问题；

## 数据结构与模型
- 固定结构数据优先使用明确模型定义，例如 `dataclass`、`TypedDict`、类对象；
- 动态字典只用于确有必要的宽松场景；
- 字段名必须稳定，禁止同一概念多种键名混用；
- 数据结构层级不得无节制嵌套；
- 核心结构必须可序列化、可打印、可调试；
- 外部输入数据必须先校验再使用；
- 数据模型必须体现业务语义，而不是只图省事；

## 错误处理
- 所有 `IO`、解析、网络请求、外部命令调用必须显式处理异常；
- 报错信息必须包含足够的定位上下文；
- 禁止空 `except`；
- 禁止直接吞掉异常而不处理；
- 捕获异常时必须优先捕获具体异常类型，而不是一把抓 `Exception`；
- 无法在当前层处理的异常应保留上下文后继续抛出；
- 能自动修复的异常应明确归类为确定性修复；
- 非确定性异常不得伪装为成功；
- 错误处理逻辑必须与主流程职责清晰分离；
- 对用户输出的报错必须可理解，对日志输出的报错必须可排查；

## 日志规范
- 运行状态、关键分支、失败原因必须通过日志或结构化输出体现；
- 日志必须包含足够上下文，但不得泄露敏感信息；
- 禁止滥用 `print` 作为正式日志方案；
- 正式项目应统一使用 `logging` 或项目统一日志组件；
- 日志级别必须语义正确，例如 `debug`、`info`、`warning`、`error`、`exception`；
- 高频路径不得输出噪声日志；
- 错误日志必须能帮助快速定位问题；
- 日志消息必须简洁明确，避免无意义废话；
- 结构化任务输出优先使用 `JSON` 或可稳定解析文本；

## 输入输出与接口
- 输入输出格式必须稳定；
- 对外接口字段名必须固定，避免隐式字段漂移；
- 所有外部输入必须校验合法性；
- 对外返回内容必须可预测、可解析、可兼容；
- 命令行参数必须提供帮助说明；
- 命令行参数命名必须清晰一致；
- 脚本退出码必须有明确语义；
- 成功路径返回 `0`，失败路径必须返回非零退出码；
- 重要输出必须支持自动化消费，不得只面向人类肉眼阅读；
- 向后不兼容变更必须有记录、有说明、有迁移预期；

## 文件与路径处理
- 文件路径处理优先使用 `pathlib`；
- 路径拼接不得依赖硬编码字符串拼接；
- 读写文件前必须校验路径存在性、权限或可创建性；
- 文件写入必须考虑覆盖行为、原子性和失败回滚；
- 涉及删除、移动、覆盖等操作时必须显式确认目标行为；
- 临时文件必须有明确生命周期；
- 跨平台路径行为必须提前考虑，禁止默认只按单一系统习惯编写；

## 配置管理
- 配置必须与代码逻辑分离；
- 可变配置不得硬编码在业务代码中；
- 默认配置必须安全、保守、可解释；
- 配置项命名必须稳定；
- 配置加载失败时必须给出清晰错误信息；
- 敏感配置不得硬编码提交到仓库；
- 环境变量、配置文件、命令行参数的优先级必须明确；
- 配置结构复杂时必须提供统一读取入口；

## 测试与校验
- 脚本类代码至少提供最小可执行校验路径；
- 关键逻辑必须具备最小单元测试覆盖；
- 复杂分支必须覆盖正常路径、边界路径和失败路径；
- 测试名称必须表达场景和预期；
- 测试必须稳定、可重复，禁止依赖随机外部状态；
- 测试不得污染真实环境；
- 需要外部依赖的测试应尽量隔离或 mock；
- 回归 bug 应补充对应测试；
- 关键路径命令必须支持非零退出码；
- 重要功能必须至少存在一种自动化校验方式；

## 并发与异步
- 只有在确有收益时才使用异步或并发；
- 异步函数边界必须清晰，禁止同步异步混乱穿插；
- 并发逻辑必须显式考虑共享状态安全；
- 不得在未理解线程、进程、协程语义的前提下随意并发化；
- 超时、取消、重试、资源释放必须设计清楚；
- 并发失败必须可观测、可回收、可定位；

## 性能要求
- 优先保证正确性与清晰性，再进行性能优化；
- 性能优化必须基于明确瓶颈，而不是猜测；
- 高频路径中的重复计算、重复解析、重复 `IO` 必须避免；
- 大数据处理必须控制内存占用；
- 批量处理优先使用流式或分段方式；
- 缓存必须有明确边界、失效策略和一致性预期；
- 不得为了微小收益牺牲整体可读性；

## 安全要求
- 禁止硬编码密码、密钥、令牌和敏感凭据；
- 日志、异常、调试输出中不得泄露敏感信息；
- 外部输入必须做合法性校验和必要清洗；
- 执行外部命令时必须防止命令注入；
- 文件操作必须防止路径穿越等风险；
- 反序列化、动态执行、模板渲染等高风险能力必须谨慎使用；
- 任何具破坏性的操作都必须显式受控；
- 默认策略必须偏安全；

## 工程可维护性
- 模块职责必须清晰，目录结构必须可预测；
- 通用能力应沉淀复用，避免散落重复实现；
- 代码评审必须重点关注正确性、边界、兼容性、可读性和风险；
- 技术债必须可见，不得长期隐藏在实现细节中；
- 变更必须控制影响范围；
- 大改动必须优先分步落地，避免一次性引入过多不确定性；
- 所有关键实现都应尽量做到可定位、可回滚、可审计；

## 禁止事项
- 禁止未处理异常的关键 `IO` 操作；
- 禁止空 `except`；
- 禁止 `from xxx import *`；
- 禁止使用可变对象作为默认参数；
- 禁止在核心逻辑中硬编码敏感信息；
- 禁止通过复制粘贴堆积重复逻辑；
- 禁止无意义注释和过期注释；
- 禁止命名含糊不清；
- 禁止返回不稳定的数据结构；
- 禁止将调试代码、临时 hack、无用代码长期保留在主分支；
